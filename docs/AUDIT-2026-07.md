# Part 1 — Security


## Executive summary

Sixteen verified findings survived triage. They cluster into three root-cause families plus a band of CI/config hygiene gaps. The single most important issue is a **critical privilege-escalation** in the RBAC demo-mode heuristic, followed by a **high SSRF-to-cloud-metadata** in federated-peer fetches. Everything else is medium or low.

Two systemic root causes account for five of the top six findings and should be fixed once, at the source:

1. **Demo-mode is inferred from one legacy env var (`OIDC_ISSUER_URL`).** Every modern auth path — named OIDC providers, SAML, OAuth2, magic-link — leaves that var unset, so `isDemo` is `true` in real deployments and *every authenticated user is silently granted full manager+pmo+admin*, bypassing role claims and the strong-auth (WebAuthn) gate. The same flawed check is copy-pasted into the dev-mode impersonation gate. **Fix: one shared `isDemoAuth()` helper computed from "no real auth configured at all," used at both call sites.**

2. **Outbound fetches that skip the DNS-resolving egress guard.** Federation and OIDC discovery/token-exchange call the global `fetch()` (or the literal-only `assertSafeOutboundUrl`) instead of `safeFetch`/`assertEgressAllowed`. Because config-time validation never resolves DNS, an attacker-registered host resolving to `169.254.169.254` reaches cloud IMDS — leaking IAM creds, the peer bearer token, and (for OIDC) the `client_secret` — and the data-residency egress gate is bypassed on these hops. **Fix: route both through `safeFetch`, exactly as `webhooks.ts` already does.**

The remaining medium issues are independent: a silent decrypt-to-empty-then-overwrite path that can destroy the vault/rate-card stores on a key rotation; an unvalidated governance `when` predicate on `PATCH /settings` that lets an admin persist a rule which 500s feature resolution for *all* users; and an unpinned/`--no-save` `tsx` install in the credential-holding retention-broker image.

### Posture — what is genuinely solid

This is a well-architected, security-conscious codebase, and several findings only exist *because* strong controls are present elsewhere and are inconsistently applied:

- **The zero-at-rest / Broker-seam design holds.** No above-seam persistence was found; the one flagged gap is a CI-guard blind spot, not a live durable leak (the `ioredis` use is an ephemeral TTL cache).
- **Egress is genuinely centralised and DNS-rebinding-aware** in `lib/egress.ts` (`safeFetch` resolves DNS, blocks link-local/metadata IPs, and enforces residency). The SSRF findings are *missed call sites*, not a missing control — the correct primitive already exists and is used on the broker, setup, and webhook hops.
- **Primary CSRF defense is intact** (SameSite=Lax cookies); the flagged case-sensitivity gap is defense-in-depth only.
- **Secrets handling is layered** (HMAC-hashed identities, KMS-fronted vault, read-time redaction), and the crypto findings are error-handling/rekey-documentation edges, not weak primitives.
- **Supply-chain discipline is above average** (digest-pinned base images, `minimumReleaseAge`, Dependabot). The supply-chain findings are consistency gaps against that same high bar.

Net: fix the two systemic root causes (RBAC demo-mode + egress-guard parity) and the codebase is in strong shape.

# Part 2 — Clean code


## Overview
28 verified findings, all confirmed real. Every one is a maintainability/consistency cleanup — **no correctness or security defect survived verification**. The dominant theme is *duplication of an existing helper*: in almost every case a canonical abstraction already exists (`getJson`, `sendJson`, `actorForAudit`, `parseCommaSet`, `emitRegistry`, `walkFiles`, `StatCard`, `updateSettings`) and is only partially adopted, so the same shape is re-inlined 5–24 times alongside it. Two findings carry a small behavioral upside beyond DRY (read-side `getJson` surfaces real server errors + step-up; broker-source restores egress/residency coverage); the rest are pure boilerplate reduction and drift prevention.

Grouped into six themes: **API routes**, **API lib**, **API subsystems**, **SPA components**, **SPA lib**, **Catalogue/CI scripts**.

## Top picks (do these first)
1. **Read-side `getJson` adoption** — highest payoff. ~13 GET helpers re-inlined; collapsing them also fixes a real UX gap (PMO/admin 403s currently show a bare status code, and none detect `step_up_required`). `artifacts/omniproject/src/lib/setup.ts:41`.
2. **Write-side `sendJson` extension + adoption** — ~12 hand-rolled mutation fetches plus 3 private near-dupes; a two-line extension (204 tolerance + `fallback`) unlocks deleting all of them. `artifacts/omniproject/src/lib/api.ts:15`.
3. **`settingsCollectionRouter` factory** — ~180 lines across 7 route files reduce to one call each, and absorbs the 13× `SettingsValidationError` catch too. `artifacts/api-server/src/routes/views.ts:13`.
4. **Themed Recharts primitives + `ReportEmpty` + `ProjectHistoryChart`** — the SPA reports layer is the widest surface (chart scaffolding in 16 files, empty-state in 24, chart family of 5); three small shared components centralise theming so a token change stops being a 16-file edit. `artifacts/omniproject/src/components/reports/CumulativeFlow.tsx:31`.
5. **`broker-source` through `safeFetch`** — the only outbound path bypassing the centralised egress/SSRF/residency guard; low live risk (operator-supplied URL) but worth doing to keep the posture uniform. `artifacts/api-server/src/history/broker-source.ts:25`.
6. **Single canonical-JSON serializer** — two divergent implementations feed security-critical, must-stay-reproducible hashes/MACs; consolidate now as cheap insurance against silent drift. `artifacts/api-server/src/lib/snapshot.ts:40`.
