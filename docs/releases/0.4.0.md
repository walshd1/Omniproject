# OmniProject 0.4.0 — Modelling, history & test maturity

**Release date:** 2026-06-25
**Type:** Minor (pre-1.0). **No breaking API changes.**
**Tag:** `0.4.0` (un-prefixed) · compare `0.3.0...0.4.0`

> This is the GitHub Release body for 0.4.0. The canonical, versioned record is
> [CHANGELOG.md](../../CHANGELOG.md); this page is the human-readable companion
> with upgrade notes and an honest maturity table.

---

## TL;DR

0.4.0 turns OmniProject from "a hardened overlay" into "a hardened overlay you
can **model and reason about over time** — and **trust the test suite of**." It
adds:

- A stateless **Exploration** workspace (`/explore`): snapshots → trends, a
  **What-If** sandbox, and **cross-system dependency links by hash**.
- An opt-in, gated **time-travel** preview (history/replay) behind a secure,
  warranty-disclaimed **logging-sync** egress.
- **Real test suites with enforced CI coverage gates** where the SPA had none —
  ~640 automated tests total — plus a **36-finding forensic security pass**.

Everything new stays **broker-agnostic and above the seam**, and every surface is
tagged **Stable / Beta / Experimental** so nothing oversells its maturity.

## Maturity at a glance

| Area | Status | What that means here |
|------|--------|----------------------|
| Test suites + coverage gates | **Stable** | Tested, CI-enforced, production-intended. |
| Security fixes (36-finding pass) | **Stable** | Fixed and independently re-verified. |
| Exploration mode (`/explore`) | **Beta** | Functional and tested, new, not yet hardened by real-world use. |
| Time-travel (history/replay) | **Experimental** | Complete and tested *at the seam/contract*; **unproven end-to-end** against a live logging server. Preview. |

**Legend.** **Stable** = tested and production-intended. **Beta** = functional and
tested but new. **Experimental** = complete and tested at the contract, but the
end-to-end path against real external systems is unproven — treat as a preview.

## What's new

### 1. Comprehensive test suites + enforced coverage gates *(Stable)*

The SPA previously had **no** automated tests. It now has a full suite
(**Vitest + React Testing Library + jsdom, ~400 tests**), and the gateway suite
grew to **~240 tests**. CI enforces coverage ratchets on both:

- Gateway: **`c8`**, ~84% lines (branches/functions/statements gated too).
- SPA: **Vitest v8**, ~88% lines / ~84% branches.

**Honest caveats** (also in [docs/TESTING.md](../TESTING.md)):

- SPA **function** coverage is ~64% — many inline event handlers aren't
  individually invoked.
- Several flows are **render-tested, not interaction-tested** (drag-drop
  optimistic moves, full-page journeys).
- The axe-core a11y job covers the **core routes only**.

### 2. Exploration mode — `/explore` *(Beta)*

A deliberately **distinct, "NOT LIVE DATA" surface** for modelling, kept separate
from the live app (and pop-out-able) so a modelled or historical figure can never
be mistaken for production. Everything here is **client-side and
session-volatile** — the gateway stays **stateless and zero-data-at-rest**. You
**download to keep**, or the work is discarded at session end. Full guide:
[docs/EXPLORATION.md](../EXPLORATION.md).

- **Portfolio snapshots → trends** — capture the live read-model at 1..N points;
  export/import a JSON bundle for durable multi-month trends. Badged `captured`.
- **Auto-snapshot schedule** — capture on a user-defined interval until an
  end date/time. *Limitation:* runs **only while the tab is open** — a durable
  overnight cadence is the n8n historian's job, not the browser's.
- **What-If sandbox** — a volatile fork of the portfolio with coarse
  completion / schedule / budget / blocker levers and baseline-vs-scenario deltas.
  Can be **based on any captured snapshot**, and "capture as snapshot" feeds it
  back into trends. *Limitation:* portfolio-level, coarse levers — a modelling
  aid, **not a planner**, and explicitly not a PM system by the back door.
- **Cross-system dependency links by hash** — mark an event in one system as
  dependent on an event in another; OmniProject stores **two SHA-256 fingerprints
  + minimal refs only** (never content — guarded by an anti-creep test) and
  surfaces the link with **live drift detection**. *Limitation:* drift recomputes
  only for endpoints whose projects are currently loaded.

### 3. Time-travel *(Experimental / preview)*

An opt-in, **gated** history/replay capability. What's **done and tested at the
seam**: the contract, the admin-only + SSRF-validated + warranty-acknowledged
**logging-sync** opt-in, the `timeTravel` capability flag, the `Broker.replay`
method, and the gated `GET /history/replay` (returns **409 until enabled**).

What's **unproven end-to-end** (hence Experimental):

- `DemoBroker.replay` returns **synthesised `sample` data**.
- The n8n historian/replay blueprint
  ([omniproject-time-travel.json](../../artifacts/n8n-blueprints/omniproject-time-travel.json))
  is a **template** (`active: false`) — there is **no integration test against a
  live logging server** yet.
- Forward time-travel is a **`projected` model, never fact**.

Off by default. Full guide and threat notes: [docs/TIME-TRAVEL.md](../TIME-TRAVEL.md).

## Security

- **36-finding forensic review — all fixed and re-verified** *(Stable)*. Highlights:
  a **critical production fail-fast** on a default/empty `SESSION_SECRET`; a
  contributor gate on `POST /broker/command`; an **SSRF guard** on admin-set
  outbound URLs (including the IPv4-mapped-IPv6 metadata-endpoint bypass); and no
  longer leaking upstream backend response bodies in error messages.
- **`GET /settings` no longer leaks webhook signing secrets** *(Stable)*. The read
  endpoint — reachable by **any** authenticated session, including read-only API
  tokens — now masks webhook secrets. *Known limitation:* it still returns
  non-secret config (broker / issuer / logging-sync URLs) to any authenticated
  session; tightening that to admin-only is a tracked follow-up.

---

## Changed

- **Opt-in state-history egress ("logging sync")** *(Experimental)* — off by
  default; admin-only; the destination URL is **SSRF-validated**; enabling
  **requires an explicit acknowledgement that egressed data leaves OmniProject's
  warranty** (the same trust class as the existing OData / Power-BI feeds). This is
  the **single deliberate relaxation** of the "nothing leaves" posture —
  OmniProject still stores nothing itself.

---

## Architecture note — still broker-agnostic, still above the seam

Every new capability lands **above the broker seam**: the new `replay` operation
is a method on the `Broker` **interface**, all n8n specifics stay confined to
`N8nBroker`, and the **architecture-guard**, **broker-conformance**, and
**contract-coverage** tests all pass. n8n is the broker that ships today; if you
run something else, you implement one class and nothing above the seam moves.

## Upgrade notes

- **No breaking changes.** Drop-in over 0.3.0.
- **No new required configuration.** Exploration mode is on by default but is
  entirely client-side and stores nothing server-side.
- **Time-travel and logging-sync are opt-in and off by default.** To preview
  time-travel you must, as an admin, set a logging-sync URL **and** acknowledge the
  warranty disclaimer; until then `GET /history/replay` returns **409**. Treat it
  as a preview — there is no live-server integration test yet.
- **Contract regenerated.** If you consume the generated client, pull the updated
  `lib/api-zod` / `lib/api-client-react` (new: `/history/replay`, `HistoryState`,
  `Capabilities.timeTravel`, `Settings.loggingSync`). The OpenAPI spec is the
  source of truth; CI enforces codegen drift.
